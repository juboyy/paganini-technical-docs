Stages 11-15: Execução, QA & Review Humana

Stage 11: Developer (Implementation)​

STAGE 11 — OWNER: COSMO (via Codex CLI)Developer (Implementation)

Implementação de código delegada ao Codex CLI (gpt-5.3-codex) com orquestração multi-agente e aderência a TDD absoluto (testes falhando primeiro). OraCLI não escreve código diretamente.

Ferramentas: Codex CLI (gpt-5.3-codex), GitNexus (contexto), VS Code/Cursor
Artefatos: Código-fonte, Testes Unitários/Integração

Codex Multi-Agent Roles:

SurfaceCapability
CLIMulti-agent parallelism, MCP, terminal
IDEVS Code/Cursor extension
WebCloud async tasks (ChatGPT)
Regra de Codex

Qualquer implementação > 5 linhas de código DEVE ir pelo Codex. OraCLI escrevendo .tsx, .ts, .py diretamente é uma violação do SOUL. Exceção: Micro tier (< 2 min, single-line fix).

Stage 12: Code Review​

STAGE 12 — OWNER: COSMOCode Review

Revisão automatizada e assistida por IA. Verifica padrões, segurança, performance e aderência ao spec.

Ferramentas: Gemini (análise), SonarQube (integração), lint/format scripts
Artefatos: review.md, Sugestões de Refatoração

Checklist de Review:

  • Aderência ao spec.md e gate token
  • Sem secrets ou API keys no código
  • Testes unitários cobrindo path crítico
  • Sem breaking changes em APIs públicas
  • Performance: sem N+1 queries, sem loops bloqueantes
  • Acessibilidade (se frontend)

Stage 13: QA & Verification (AXI)​

STAGE 13 — OWNER: COSMO (via AXI)QA & Verification

Testes rigorosos incluindo E2E, performance e segurança. Validação visual via Chrome DevTools AXI (Accessibility Tree Inspector) — eficiência máxima de tokens.

Ferramentas: AXI (browser automation), Playwright (E2E), Scripts de Teste
Artefatos: Relatório de Teste, Screenshots/Vídeos

Por que AXI e não screenshots:

MétodoTokens/páginaInformação
AXI (accessibility tree)~800Estrutura semântica completa
PinchTab snap~800Refs interativos
Screenshot (base64)~5,000-10,000Pixel data, sem semântica
Regra de PR

Todo PR deve ter um vídeo demo linkado mostrando o teste/feature funcionando. PR sem vídeo = PR incompleto. Gravação via browser.py record-start → ações → browser.py record-stop.

Stage 14: Deploy Preview​

STAGE 14 — OWNER: COSMODeploy Preview

Deploy em ambiente de preview para validação por stakeholders e QA antes da aprovação final.

Ferramentas: Vercel Preview, GitHub Actions
Artefatos: URL de Preview, Log de Deploy

# Deploy preview (sem --prod)
cd /tmp/agent-orchestrator && npx vercel deploy --token "$DEPLOY_TOKEN" --yes
# Output: https://agent-orchestrator-xxx-yyy.vercel.app (preview URL)

Stage 15: Human Review (Linear Gate)​

STAGE 15 — OWNER: 👤 João — 🔴 PRODUCTION GATEHuman Review (Linear Gate)

O Stakeholder aprova o dossiê completo no Linear. Único gate de aprovação humana para merge em produção. Não há deploy em prod sem aprovação explícita.

Ferramentas: Linear (review hub), linear-approval-poller.py (cron every 30m)
Artefatos: Status de Aprovação no Linear

Dossiê Linear inclui:

  • Link do PR no GitHub
  • Documentação técnica (spec.md, architecture.md)
  • Análise de blast radius (GitNexus)
  • Telemetria (tokens, custo, latência)
  • Vídeo demo
  • Codex prompts usados

Ver detalhes em Human Gate.
